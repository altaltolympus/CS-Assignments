Ag Comhlíonadh and Coimre

BR1: Dearadh Córais Leabaithe

Tá an córas leabaithe atá déanta agam curtha i ngíomh agus as gníomh trí bhrú chnaipe, chomh maith le hionchur analógach a thógáil ó bhraiteoirí atá nasctha leis.

Aschuirtear na sonraí sin go dtí ríomhaire atá ceangailte leis trí USB, chomh maith le bheith léirithe ar scáiléan an micro:bit. Tá aschur fuaime freisin, agus solas preabach stádais.

BR2: Bailiú sonraí

Baineann an córas úsáid as bhraiteoirí timpeallachta chun bogthasice ithreach agus léibhéal uisce a thomhais. Ansin, seolann sé na sonraí seo chuig ríomhaire thar cheangailt USB ag baint úsáid as an modúl python serial, áit a stóráiltear é i gcomhad .csv.

BR3: Ionsamhlú próiséise

Tá feidhm curtha isteach agam chun oibriú an chórais a chur chun triall. Déanann an feidhm seo, a spreagtar trí bhrú chnaipe, ionsamhlú de na léibhéal bogthaise ithreach agus léibhéal uisce ag dul in airde de réir a chéile sula dtiteann siad go náid arís.

Ardriachtanais:

Samhaltú baoil tubaiste, ionsamhlúchán "mas rud é" agus oiriúiniúchán

Sa ríomhchlár atá á rith ar an ríomhaire atá nasctha leis an micro:bit, tá córas a dhéanann measúnú ar na sonraí a fhaigheann sé agus a oibríonn amach an baol tuile, más baol beag, baol ard, nó más dócha go bhfuil tuile ar siúl cheana féin.

Chomh maith leis sin, chruthaigh mé ionsamhlú le rith ar an micro:bit a dhéanfadh aithris ar na braiteoirí ag braith léibhéal bogthaise ithreach agus léibhéal uisce ag dul in airde i ndiaidh a chéile, agus ansin ag crapadh ar ais arís.

Faoi dheireadh, tar éis luachanna a léamh ó na braiteoirí a chinntear go bhfuil siad tar éis léibhéal contúirteach a shroicheadh i comparáid le tairsigh ionsamhlaithe, déanfaidh an córas aláram a chur ag bualadh.

Roghanna Tionscadail

Agus mé ag cur tús leis an tionscadal seo, bhí roghanna iomadúla agam de thionscadal féideartha a shásódh an choimre. Roghnaigh me an treo seo ar roinnt cúiseanna, áfach.

D'fhéadfadh liom córas rabhaithe falscaí bunaithe ar bhraiteoirí teochta agus bogthaise atmaisféaracha a dhéanamh; ba rogha coitianta é seo i mo rang, ach shíl mé go mbéadh sé níos oiriúnaí obair le tuilte, mar go bhfuil tuilte i bhfad níos líonmhaire in Éireann ná mar atá falscaithe.

Bhí mé in ann córas ilchumasach samhaltú aimsire a dhéanamh freisin, rud a dtuarfadh an aimsir bunaithe ar chruinneagán de a lán fachtóirí agus ionchuir éagsúla timpeallachta, ach mheas mé go mbeadh sé seo róneamhphraiticiúil agus neamiontaofa chun tástáil a dhéanamh air faoi choinníolacha ilchineálach.

Agus é seo ar fad ar intinn agam, chinn mé go ndéanfadh mé córas rabhaithe tuile, agus go léireodh mé na sonraí uaidh i ndá bharra ar scáileán an micro:bit (barra amháin do bhogtaise ithreach agus barra eile do léibhéal uisce), a leathnódh ó chlé go deis an scáileáin de réir mar a ardódh na athróga comhfhreagrach, agus a dtarraingeodh siar de réir mar a ísleodh siad.

Teicneolaíochtaí agus Bogearraí in Úsáid

Ba iad na braiteoirí a roghnaigh mé le húsáid, agus de bharr sin na luachanna a bheadh mé á dtomhas, braiteoirí bogthaise ithreach agus léibhéal uisce. Bhí cúiseanna agam dó seo: ar an gcéad dul síos, tá baint láidir ag na luachanna seo le titim báistí agus toisceanna timpeallachta eile a chuireann le tuilte. Ar an dara dul síos, bhí braiteoirí do na hathróga seo ar fáil dom go héasca. Ar an tríú dul síos, beadh sé i bhfad níos praiticiúla na braiteoirí seo a chur chun triall faoi choinníollacha éágsúla ná braiteoirí eile a d'fhéadfadh liom iad a úsáid freisin, mar gur féidir na toisceanna a thomhaiseann siad seo a ionramháil go héasca.

Roghnaigh mé chun na luachanna a bhailítear ó na braiteoirí a chur in iúil trí I did this as an alternative to relying on the micro:bit's inbuilt text display function, which I felt was not only too slow to provide information to keep up with the rate at which the environment changed, but would stop everything else from running while it was working, including time-sensitive things like the alarm.

I then decided to store the data sent from the micro:bit into a .csv file, rather than an online database service such as ThingSpeak; firstly because I have more experience with using Python's serial module than with such services, but also to account for the poor reliability of internet connection in my school building.

In the making of the embedded system and model, I used Thonny and Microsoft MakeCode for micro:bit as IDEs. When I was reporting on the project, I used Notepad + + to write the HTML and CSS for this page.


End user needs

This project would be useful to a variety of interests. Among these are people living in flood-prone areas, looking to know when a flood is imminent in order to adequately prepare; this is where features like the alarm come in handy. As well as this, there are those who would study the weather in a given area and would find records of the data I am storing useful to that end. In this case, the status bars will be invaluable, allowing the user to see the state of the area with only a passing glance.

Testing

I carried out testing on the embedded system, data collection system and model each separately before they were ready to be combined. I carried out unit testing as I went, testing the embedded system using the MakeCode micro:bit simulator, and testing the data collection and model in the command line.

Problems

One of the problems I encountered along the way was that, when I connected the water level sensor to the micro:bit, it always filled a part of the status bar I was using to monitor the readings and caused a line to appear down the middle of the micro:bit's screen.

This was a problem, not only because I did not expect this and therefore was not sure if this was affecting the sensor's readings that were being collected and stored, but also because being able to clearly read the screen was important for me to easily make sure the readings from the sensor were working, and the inaccurate bar and vertical line were making this impossible.

In an effort to fix this, I examined the wiring connections of the sensor until I noticed that the end connected to the micro:bit was larger than that of the other sensor. Experimentally, I reversed the connection so each end of the wire was where the other one had been, and after doing this the sensor worked as expected, showing the proper value of 0 when not in water and without the vertical line that interfered with my readings.

Model

The model that I implemented makes use of the data being collected as soon as it is sent over to the computer. The soil humidity and water level are both compared with simulated thresholds, representing how high these values would have to get for them to be dangerous. This risk factor is stored alongside the collected data in the same .csv file.

When the situation is normal, the flood risk will be displayed as "None". When one variable reaches a dangerous level, however, it will instead be "Risk". If both are dangerously high, the system will assume a flood is already happening, and the model will read that a flood is "In progress". If the flood simulation is running, however, the model will respond the flood risk as "Simulation", no matter how high the levels.

Evaluation

In conclusion, the artefact I have created allows the user to closely monitor an area for any signs of an approaching flood, which is exactly what is required of it to do. It can both display this information itself and send it to another computer for storage, which can allow for deeper analysis into patterns and trends in that area.

Even so, there are a few more ways this project could have been improved. One of these is a more in-depth model that takes into account whether water level and soil humidity are rising or falling from their previous values. By analysing the levels from more than one reading, it would be possible to have a much more accurate prediction of whether a flood would begin.

Another way to improve the artefact would have been to allow for wireless transmission of data from the micro:bit to the computer that stores it, whether by way of Bluetooth connection or an online database service like ThingSpeak. I ultimately decided against this for reasons I explained earlier, but this would create the opportunity to use this system in a wider variety of environments where a wired connection would not be practical.
