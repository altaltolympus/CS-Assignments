Meeting The Brief

BR1: Embedded System Design

The embedded system I made is activated and deactivated by the press of a button, as well as taking analogue input from connected environmental sensors.

This data is then outputted to a USB-connected computer, as well as being displayed as bars on the micro:bit's screen. There is also audio output, and a blinking status light.

BR2: Data collection

The system uses an environmental sensor to measure both soil humidity and water level. It then sends this data to a computer over USB connection making use of the serial python module, where it is stored in a .csv file.

BR3: Process simulation

I have implemented a function to test the operation of the system. This function, triggered by button press, simulates water level and soil humidity readings gradually becoming very high before falling back down again.

Advanced Requirements:

Disaster risk modelling, "what-if" simulation and adaptation

The program running on the computer connected to the micro:bit contains a system which would assess the environmental data it recieved and determine the risk of flooding, whether there was little risk, high risk, or most likely already a flood in progress.

As well as this, I created a simulation to be run on the micro:bit which would imitate the sensors detecting humidity and water level increasing in tandem before receding again.

Finally, the system will, upon reading values from the sensor which are determined to be at dangerous levels by comparison to simulated thresholds, sound an alarm once.

Planning & Design

Project Options

As I started this project, there were multiple directions I could have taken that would have satisfied the brief. However, I eventually chose the project that I did for a number of reasons.

I could have made a wildfire early-warning system based on a temperature and atmospheric humidity sensor, a popular idea in my class; however, I felt it would be more appropriate to work with flooding, since floods vastly outnumber wildfires in Ireland.

I could also have made an all-purpose weather modelling system, which would predict the weather based on an agglomeration of many different environmental inputs, but I deemed this to be too impractical and unreliable to test under varied conditions.

With all this in mind, I decided I would make the flood warning system and display the data from it in two bars on the micro:bit's screen (one for soil humidity and one for water level), which would extend from the left to the right of the screen as the corresponding variables grew higher, and retract as these decreased.

Technologies And Software Used

The sensors I decided to use, and therefore the variables I would be measuring, were water level and soil humidity. I chose these for a number of reasons. Firstly, these metrics strongly correlate with rainfall and other things which cause flooding. Secondly, sensors for these values were easily available to me. Thirdly, it would be much more practical to test these under varied conditions than other sensors which could also have been used, as the conditions these detect can easily be changed artificially.

I did this as an alternative to relying on the micro:bit's inbuilt text display function, which I felt was not only too slow to provide information to keep up with the rate at which the environment changed, but would stop everything else from running while it was working, including time-sensitive things like the alarm.

I then decided to store the data sent from the micro:bit into a .csv file, rather than an online database service such as ThingSpeak; firstly because I have more experience with using Python's serial module than with such services, but also to account for the poor reliability of internet connection in my school building.

In the making of the embedded system and model, I used Thonny and Microsoft MakeCode for micro:bit as IDEs. When I was reporting on the project, I used Notepad + + to write the HTML and CSS for this page.


End user needs

This project would be useful to a variety of interests. Among these are people living in flood-prone areas, looking to know when a flood is imminent in order to adequately prepare; this is where features like the alarm come in handy. As well as this, there are those who would study the weather in a given area and would find records of the data I am storing useful to that end. In this case, the status bars will be invaluable, allowing the user to see the state of the area with only a passing glance.

I carried out unit testing as I went, testing the embedded system using the MakeCode micro:bit simulator, and testing the data collection and model in the command line.

Problems

One of the problems I encountered along the way was that, when I connected the water level sensor to the micro:bit, it always filled a part of the status bar I was using to monitor the readings and caused a line to appear down the middle of the micro:bit's screen.

This was a problem, not only because I did not expect this and therefore was not sure if this was affecting the sensor's readings that were being collected and stored, but also because being able to clearly read the screen was important for me to easily make sure the readings from the sensor were working, and the inaccurate bar and vertical line were making this impossible.

In an effort to fix this, I examined the wiring connections of the sensor until I noticed that the end connected to the micro:bit was larger than that of the other sensor. Experimentally, I reversed the connection so each end of the wire was where the other one had been, and after doing this the sensor worked as expected, showing the proper value of 0 when not in water and without the vertical line that interfered with my readings.

Model

The model that I implemented makes use of the data being collected as soon as it is sent over to the computer. The soil humidity and water level are both compared with simulated thresholds, representing how high these values would have to get for them to be dangerous. This risk factor is stored alongside the collected data in the same .csv file.

When the situation is normal, the flood risk will be displayed as "None". When one variable reaches a dangerous level, however, it will instead be "Risk". If both are dangerously high, the system will assume a flood is already happening, and the model will read that a flood is "In progress". If the flood simulation is running, however, the model will respond the flood risk as "Simulation", no matter how high the levels.
