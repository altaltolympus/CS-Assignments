Meeting The Brief

BR1: Embedded System Design

The embedded system I made is activated and deactivated by the press of a button, as well as taking analogue input from connected environmental sensors.

This data is then outputted to a USB-connected computer, as well as being displayed as bars on the micro:bit's screen. There is also audio output, and a blinking status light.

BR2: Data collection

The system uses an environmental sensor to measure both soil humidity and water level. It then sends this data to a computer over USB connection making use of the serial python module, where it is stored in a .csv file.

BR3: Process simulation

I have implemented a function to test the operation of the system. This function, triggered by button press, simulates water level and soil humidity readings gradually becoming very high before falling back down again.

Advanced Requirements:

Disaster risk modelling, "what-if" simulation and adaptation

The program running on the computer connected to the micro:bit contains a system which would assess the environmental data it recieved and determine the risk of flooding, whether there was little risk, high risk, or most likely already a flood in progress.

As well as this, I created a simulation to be run on the micro:bit which would imitate the sensors detecting humidity and water level increasing in tandem before receding again.

Finally, the system will, upon reading values from the sensor which are determined to be at dangerous levels by comparison to simulated thresholds, sound an alarm once.

Investigation

Planning & Design

As I started this project, there were multiple directions I could have taken that would have satisfied the brief. However, I eventually chose the project that I did for a number of reasons.

I could have made a wildfire early-warning system based on a temperature and atmospheric humidity sensor, a popular idea in my class; however, I felt it would be more appropriate to work with flooding, since floods vastly outnumber wildfires in Ireland.

I could also have made an all-purpose weather modelling system, which would predict the weather based on an agglomeration of many different environmental inputs, but I deemed this to be too impractical and unreliable to test under varied conditions.

With all this in mind, I decided I would make the flood warning system and display the data from it in two bars on the micro:bit's screen (one for soil humidity and one for water level), which would extend from the left to the right of the screen as the corresponding variables grew higher, and retract as these decreased.

I did this as an alternative to relying on the micro:bit's inbuilt text display function, which I felt was not only too slow to provide information to keep up with the rate at which the environment changed, but would stop everything else from running while it was working, including time-sensitive things like the alarm.

I then decided to store the data sent from the micro:bit into a .csv file, rather than an online database service such as ThingSpeak; firstly because I have more experience with using Python's serial module than with such services, but also to account for the poor reliability of internet connection in my school building.
